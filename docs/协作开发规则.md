# 开发规则

## 1. 分支策略

- `main`：稳定分支，只用于合并与发布，禁止直接在 `main` 上开发。
- `feature/*`：功能分支，每个成员/每个功能点一个分支。

分支命名建议：

- `feature/姓名-模块`
- `feature/模块-功能点`

示例：

- `feature/booking-cancel`
- `feature/booking-edit`
- `feature/member-level`

## 2. 成员操作

### 2.1 第一次获取代码

```bash
git clone <仓库地址>
cd <项目目录>
```

### 2.2 开始开发（创建分支）

```bash
git checkout main
git pull origin main
git checkout -b feature/xxx
```

### 2.3 提交与推送

```bash
git add .
git commit -m "完成xxx：简要说明"
git push -u origin feature/xxx
```

建议：

- 每次提交只做一件事（便于回滚与定位问题）。
- 每次开始开发前、准备推送前都执行一次 `git pull origin main`（减少冲突）。

## 3. 高冲突文件约定（必须遵守）

以下文件/目录容易冲突，原则上避免多人同时修改：

- `db_hotel_ssm.sql`（数据库脚本）
- `src/config/db.properties`（数据库连接配置）
- `src/config/spring/*.xml`（Spring 配置）
- `src/config/springmvc/*.xml`（Spring MVC 配置）

## 4. 合并规则（管理员执行）

- `feature/*` 分支测试没有问题再合并到 `main`。

如果发生冲突，解决冲突确认无误后再合并。

## 5. 推荐分工落地方式

- 每个模块在 `docs/开发差距与待办清单.md` 里维护“负责人/状态/链接（分支名）”。
- 完成某个任务需要同步更新 `docs/开发差距与待办清单.md`
